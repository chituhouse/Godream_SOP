## 问题概述
- Electron 前端在 dev 模式出现 `getSnapshot should be cached` 与 `Maximum update depth exceeded`，导致界面白屏。
- 根因：体验卡片与 Formily 表单同时写入引用 store，selector 返回新引用；`ReferencePicker` 内部 effect 与 store 写入形成闭环。

## 关键改动
1. **单一真源**：保留 Formily 表单为引用数据唯一写入入口；首页“参考资源体验”卡片改为只读预览，不再调用 controller 更新 store。
2. **ReferencePicker 调整**：取消对全局 store hover 状态的依赖，使用组件内 `useState`；store 写入前增加等值守卫，防止重复更新；Effect 依赖统一通过 `useMemo/useCallback` 缓存。
3. **Zustand 订阅稳定化**：App 任务列表改为原子 selector + `useMemo` 派生，避免生成新引用；引用 controller 中保留 schema 签名守卫。

## 背景
- Electron 渲染层启用 `nodeIntegration:false / sandbox:true / contextIsolation:true`，禁止直接访问 Node 内建模块。
- `seedream-client/src/utils/schema-loader.ts` 仍使用 `node:fs`、`node:path` 读取 Formily Schema，导致运行时抛出 “Dynamic require of 'fs' is not supported”，Electron 窗口白屏。

## 近期问题
- Formily 组件导入已修正（Card → antd，InputNumber → NumberPicker），白屏转由 `fs` 动态 require 触发。
- 目前渲染层内仅 `schema-loader.ts` 命中 Node 依赖，其他文件未检测到同类引用。

## 修复规划
1. **迁移 Schema 读取到主进程 IPC**：在 `electron/main/ipc` 新增 `schema:load` 等 handler，使用 Node `fs` 读取 `APP_ROOT/schemas` 下的文件。
2. **预加载桥接**：在 `electron/preload/index.ts` 暴露 `window.gemeng.schema` API，通过 `ipcRenderer.invoke` 调用主进程逻辑。
3. **前端调用重构**：`schema-loader.ts` 改为调用 `window.gemeng.schema.load`，移除 `node:fs`、`node:path`，保留缓存与 Formily 映射逻辑。
4. **验证与测试**：`npm run dev` 确认渲染层不再报错；如存在相关单测/Vitest，用新的 API 适配。

## 后续加固建议
- 阶段性执行渲染层 Node 依赖扫描脚本（rg 命令）防回归。
- 在 ESLint 增加 `no-restricted-imports` 规则，禁止渲染层 import `node:*`、`fs` 等模块（待主管确认后执行）。
